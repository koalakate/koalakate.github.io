## CRITICAL ISSUES

### [CRITICAL-1] Carousel auto-play stops after first step

**Location:** `src/components/offering/converter-carousel.tsx:48-58`  
**Dimension:** Robustness / Dead code logic  

**Problem:** `goTo(n)` sets a one-shot `setTimeout` that fires once after 4.5s and runs
`setCurrent(prev => (prev + 1) % STEPS.length)`. This correctly moves 0→1. But the `current`
state change does NOT trigger another call to `goTo`, so no new timer is set. The carousel
advances once and permanently stops at step 1. Steps 2 and 0 are never reached via auto-play.

**Fix:** Replace the progress-bar useEffect with a self-resetting timer on `current`:

```tsx
// REMOVE the existing goTo + first useEffect entirely.
// REPLACE second useEffect with:
useEffect(() => {
  const fill = fillRefs.current[current];
  if (fill) {
    fill.style.transition = "none";
    fill.style.width = "0%";
    requestAnimationFrame(() => {
      fill.style.transition = `width ${DURATION}ms linear`;
      fill.style.width = "100%";
    });
  }
  const timer = setTimeout(() => {
    setCurrent(prev => (prev + 1) % STEPS.length);
  }, DURATION);
  return () => clearTimeout(timer);
}, [current]);

// The goTo for manual clicks becomes:
const goTo = useCallback((n: number) => {
  setCurrent(n);
}, []);

// Remove timerRef entirely.
```

### [CRITICAL-2] No mobile navigation

**Location:** `src/components/nav.tsx:25`  
**Dimension:** Architecture / UX  

**Problem:** Nav links are wrapped in `className="hidden md:flex"`. Below 768px, only the
logo and the CTA button are visible. There is no hamburger menu, no slide-out drawer, nothing.
Mobile users cannot navigate to Migration Paths, Platform, Guides, or Pricing sections.

**Visual evidence:** Screenshot `09-mobile.png` — nav bar shows only logo + "Run Analyzer" button.

**Fix:** Add a mobile menu (hamburger → dropdown or drawer). Minimum viable: a `<details>/<summary>` toggle or a state-controlled `<div>` with the links, hidden on desktop, shown on mobile.

## HIGH-RISK ISSUES

### [HIGH-1] Cell 06 "Custom Path" is invisible

**Location:** `src/components/migration-paths.tsx:45-51`  
**Dimension:** Accessibility / Visual  

**Problem:** `textColor: "#ccc"` on `bg: "#F6F6F6"`. WCAG AA requires ≥4.5:1 for normal text;
this pair is ~1.2:1. The cell title "Custom Path" and "Your Target" are completely illegible.
The `arrowColor: "#ddd"` on the same background is even worse.

**Visual evidence:** Screenshot `02-migration-paths.png` — bottom-right cell content is invisible.

**Fix:** Change `textColor` to `#888` (minimum) or ideally `#555` for WCAG AA compliance.
Change `arrowColor` to `#aaa`. The "Get in touch" badge also uses `text-neutral-300` on
`#F6F6F6` — change to `text-neutral-500`.

### [HIGH-2] Five #contact links are dead — no anchor exists

**Location:** `cta.tsx:43`, `engagement-model.tsx:27,38`, `professional-services.tsx:41`, `footer.tsx:27`  
**Dimension:** Robustness

### [HIGH-3] All grids are fixed-column with no responsive breakpoints

**Locations:**  
- `migration-paths.tsx:79` — `grid-cols-3` (6 cells, always 3 columns)  
- `engagement-model.tsx:60` — `grid-cols-3` (pricing cards)  
- `guides.tsx:50` — `grid-cols-3` (guide cards)  
- `footer.tsx:37` — `gridTemplateColumns: "1.6fr 1fr 1fr 1fr"` (inline style, not responsive)  
- `analyzer.tsx:78`, `professional-services.tsx:19` — `grid-cols-2` (2-col rows)  
- `converter-carousel.tsx:78` — `gridTemplateColumns: "280px 1fr"` (inline style)  

**Problem:** At 390px mobile width, a 3-column grid gives ~130px per cell — cramped and
unreadable. The 2-column rows give ~195px — analyzer mockup and text both overflow.
The converter carousel's 280px step list + remaining width collapses to near-zero on mobile.

**Visual evidence:** Screenshot `10-mobile-full.png` confirms bento cells overlap/crush.

**Fix:** Add Tailwind responsive prefixes:
```tsx
// bento / pricing / guides:
className="grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 ..."

// 2-col rows:
className="grid grid-cols-1 md:grid-cols-2 ..."

// converter: wrap with flex-col on mobile
style={{ gridTemplateColumns: "1fr" }}  // mobile default
// override via responsive class or media query at md breakpoint
```

### [HIGH-4] Hero is visually sparse — large dead zone below content

**Location:** `src/components/hero.tsx:37`  
**Dimension:** Visual / UX  

**Problem:** The hero content (`max-w-xs` ≈ 288px wide, H1 + 3 lines subtext + 2 CTAs + note)
occupies roughly the top-left 30% of the section. The rest of the section is empty white space
except for the ASCII circle top-right. From the visual evidence (screenshot `01-hero.png`), the
bottom half of the hero viewport is essentially empty with only the section tag visible below.
This is below-spec — the design intent shows content filling the hero section better.

**Observed:** H1 "Analyze. / Convert. / Deploy." renders at ~3-4rem with tight width. Then
the very bottom of the hero (py-24 = 96px padding top + content ~200px + 96px bottom = ~390px
total section height). The content itself only takes ~240px, leaving significant empty space.

**Fix options:**
- Add a second column with a product screenshot/preview on the right side of the hero
- Increase `max-w-xs` to at least `max-w-sm` (384px) so subtext wraps less aggressively
- Reduce bottom padding or add a visual element (stats row, trust logos)

## MAINTAINABILITY PROBLEMS

### [MEDIUM-1] Extreme micro-font sizes — unreadable and fail WCAG

**Locations:** Across all components  
**Dimension:** Accessibility / Typography  

Most egregious instances:
| Component | Class | Computed size |
|-----------|-------|---------------|
| `analyzer.tsx:61` | `text-[0.34rem]` | ~5.4px |
| `analyzer.tsx:63` | `text-[0.31rem]` | ~5.0px |
| `migration-paths.tsx:98` | `text-[0.47rem]` | ~7.5px |
| `converter-carousel.tsx:114` | `text-[0.4rem]` | ~6.4px |
| `footer.tsx:53` | `text-[0.5rem]` | ~8px |
| `section tags` (global) | `font-size: 0.5rem` | ~8px |

WCAG SC 1.4.4 requires text to be resizable to 200% without loss. At 5-6px, text is already
below any readable threshold at 100%. The body text (`text-[0.72rem]` = ~11.5px) also falls
below the 16px minimum for mobile readability.

**Fix:** Scale up the type system. The rem values appear calibrated for a zoomed-in design
tool view. Multiply by ~1.4:
- Body/descriptions: min `0.875rem` (14px)
- Secondary/captions: min `0.75rem` (12px)
- Tags/kickers: min `0.625rem` (10px)
- Labels in mockups (decorative): acceptable as-is (they're visual art, not content)

### [MEDIUM-3] `direction: rtl` layout hack — screen reader order broken

**Location:** `src/components/offering/analyzer.tsx:78,80,117`  
**Dimension:** Accessibility / Architecture  

`direction: rtl` on the row swaps visual column order (screenshot left, text right) without
changing DOM order. Screen readers read in DOM order: the text column comes first in the DOM
(left column in `grid-cols-2`), which is correct semantically. But `direction: rtl` affects
all inherited text direction inside the grid, including `text-align`, list bullet position,
and inline layout — requiring an inner `direction: ltr` override on every child.

**Fix:** Use CSS `order` or `flex-direction: row-reverse`, which only affects visual order
without touching text direction semantics:

```tsx
// Remove direction:rtl from the grid. Keep grid-cols-2.
// On the screenshot div, add: className="... order-first md:order-none"
// or use Tailwind's flex with flex-row-reverse
```

### [MEDIUM-4] AsciiCell canvas dimensions race condition

**Location:** `src/components/ui/ascii-cell.tsx:16-18`  
**Dimension:** Robustness  

```tsx
const W = canvas.offsetWidth || canvas.width;
const H = canvas.offsetHeight || canvas.height;
```

On first render, `canvas.offsetWidth` is 0 (the element hasn't been laid out yet) because
React's `useEffect` fires synchronously after paint but before the browser completes layout
in some SSR/hydration scenarios. The fallback `canvas.width` defaults to 300 (HTML default),
not the actual container size. Result: canvas renders at 300×300 instead of the correct
container size (~45% of a 190px-min-height cell ≈ ~85px wide).

**Fix:** Add a `ResizeObserver` to re-render the canvas when its dimensions stabilize:

```tsx
useEffect(() => {
  const canvas = canvasRef.current;
  if (!canvas) return;
  const ro = new ResizeObserver(() => draw());
  ro.observe(canvas);
  return () => ro.disconnect();
}, [draw]);
```

### [MEDIUM-5] "Learn more →" in bento cells are non-functional `<span>` elements

**Location:** `src/components/migration-paths.tsx:128-135`  
**Dimension:** Robustness / Architecture  

The "Learn more →" text renders as a `<span>` with no `href`, no `onClick`, no `role`. It
looks like a link but does nothing. Screen readers ignore it as non-interactive. If migration
path pages will exist, these should be `<Link>` elements. If not yet, they should be removed
or replaced with a disabled state.

### [MEDIUM-6] Progress bar fill state logic inconsistency on manual click

**Location:** `src/components/offering/converter-carousel.tsx:122-128`  

When a user manually clicks step 2 (index 1) while on step 0, the fill bar for step 0 stays
animated at its current progress (mid-animation), not reset to 100% (past) or 0% (reset).
The JSX renders `width: i < current ? "100%" : "0%"` as the static value, but the active
animation for step 0's bar has `width: 100%` from the transition — it does not snap back to
"100% static" on current change. Minor visual glitch but noticeable.
